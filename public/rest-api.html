REST API

REST Conventions

REST (Representational State Transfer) APIs follow certain conventions to provide a standardized and consistent approach to building web services. The three main conventions are:

  1. Using the Right HTTP Verb: HTTP verbs such as GET, POST, PUT, DELETE, etc., should be used according to their intended purpose. For example:
    • GET - Retrieve data from the server.
    • POST - Create new data on the server.
    • PUT - Update existing data on the server.
    • DELETE - Remove data from the server.
  2. Ordering: APIs should follow a predictable order in their responses. For example, when fetching a collection of resources, they should be ordered in a consistent manner (e.g., ascending order by ID).
  3. Naming (Noun, Plural) that Maps to Collections in Your System: Resource endpoints should be named using plural nouns to represent collections of resources. For example:
    • /users - Collection of user resources.
    • /products - Collection of product resources.
    • /posts - Collection of post resources.

Full CRUDable REST API

A Full CRUDable REST API adheres to the conventions outlined above and supports CRUD operations (Create, Read, Update, Delete) on resources. This means:

  • Create: The API allows clients to create new resources using the POST method.
  • Read: The API provides endpoints to retrieve resource(s) using the GET method.
  • Update: The API supports updating existing resources using the PUT method.
  • Delete: The API allows the removal of resources using the DELETE method.

For example, a Full CRUDable REST API for managing a collection of books might have the following endpoints:

  • GET /books - Retrieve all books.
  • GET /books/{id} - Retrieve a specific book by ID.
  • POST /books - Create a new book.
  • PUT /books/{id} - Update an existing book by ID.
  • DELETE /books/{id} - Delete a book by ID.
